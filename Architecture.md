# AI-News 极简核心架构设计文档

本文档记录了应用经过**“Agentic LLM”（大模型智能体化）**第一性原理重构后的当前系统架构。

## 1. 核心设计哲学
放弃传统推荐系统繁杂的“向量检索”或“标签权重计算”，将推荐引擎的核心逻辑完全交给大语言模型（GLM）接管。
以**“自然语言文本”**作为一切个性化推荐、用户画像更新和新闻过滤的传递媒介。

## 2. 整体数据与控制流 (Data Flow)

整个系统由三个主要的闭环流构成：

### 2.1 新闻拉取与过滤 (Fetch & Filter)
1. `NewsFetcher` 解析已启用的 `RSSSource` 列表。
2. 将解析出的原始资讯与本地记录（`SeenNews`：通过哈希验证）进行比对，排除已经推荐过的新闻。
3. 选取最新的若干条候选新闻（含标题、来源、摘要），**连同当前用户的纯文本“结构化画像”**，发送给 `GLMService.pickBestNews`。
4. 大模型直接进行理解并做出唯一个决策，返回：
   - 选中的候选新闻索引 (`selected_index`)
   - 专门生成的阅读推荐语 (`reasoning`)
5. 系统将选中的这唯一一条候选打包成 `NewsItem` 写入 SwiftData 本地数据库。

### 2.2 用户行为反馈 (Feedback Loop)
1. 用户在前端阅读时进行交互（点赞/点踩/输入纯文本反馈意见）。
2. `ViewModels/NewsDetailViewModel` 会捕捉到交互，并将交互类型 (`FeedbackAction`) 和被操作的新闻记录存储为 `Feedback` 实体。
3. **立刻触发画像重塑**：后台调取该用户最近的 20~30 条 Feedback，连同其旧的版本画像一同发送给大模型（`GLMService.updateProfile`）。
4. 大模型阅读这些行为后，就像人类运营一样，直接吐出一份全面翻新的、结构明确的新“用户画像”（覆盖：核心偏好、近期追踪、避雷不看）。

## 3. 核心模块与代码映射

### 3.1 本地存储模型 (Models -> SwiftData)
架构追求极致的最少数据表依赖：
- **`NewsItem`**: 核心实体。记录标题、内容以及最关键的字段：`reasoning: String?`（存储 AI 给出的千人千面推荐理由）。不再挂载任何标签表。
- **`Feedback`**: 仅关联新闻 ID、行为（Like/Dislike）以及可选的文字点评。
- **`UserProfile`**: 存储模型生成的那 150 字以内的**纯自然语言画像**。
- **`SeenNews`**: 防重复引擎，记录看过的文章 URL 的 Hash。
- **`RSSSource`**: 支持的用户订阅源。

### 3.2 大模型交互控制心智 (Services/GLMService.swift)
该服务全面采用 **Function Calling** (工具调用) 进行约束，确保大模型每次都输出 100% 开发需要的字段结构。
其中主要的 System Prompts 设计如下：

- **选品引擎 (`pickNews`)**：
  约束了四个优先级（用户画像匹配度、实质性、新鲜度、多样性），且具备严格的防错规则（如：防标题党、防推荐连续相同的主题）。
- **画像分析引擎 (`updateProfile`)**：
  要求大模型必须把隐晦的点赞记录抽象成显性的 3 段结构文本：
  1. 核心偏好 (长期稳定的兴趣点)
  2. 近期追踪 (最近表现出的特定关注热点)
  3. 避雷不看 (极度反感或不感兴趣的内容特征)

### 3.3 UI展示层 (Views)
UI 层已剥离传统的“Tag 蓝色小药丸”展示组件（原 `TagScoreBarView` 已废除）。
核心的视觉抓手转为 `NewsCardView` 中对 `news.reasoning` (大语言模型推理文案) 的渲染，直接展示诸如：“💡 推荐理由：知道你最近非常关注马斯克星舰发射，这篇关于最新的测试前瞻送给你。”，打造极强的私人管家沉浸感。

## 4. 架构优势总结
- **前端极简**：只需拿 `reasoning` 进行文案展示，业务代码大幅缩水。
- **后端（本地库）极简**：没有中间数据库（如 Tag 中间表、关联表、算分浮点数列）。
- **无限迭代潜力**：用户的每次点踩都能直达 AI 意识层；系统不会产生标签污染，由于 AI 具备语义理解，如果用户点了关于“王思聪”的新闻的踩，模型能理解他是“不关心花边新闻”，而不是把“首富”这个词列为黑名单。
